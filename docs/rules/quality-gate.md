# Rule: quality-gate
## 品質保証 / Quality Gate

---

## Authority Hierarchy（全Ruleに共通）

| 優先順位 | ドキュメント | 役割 |
|----------|-------------|------|
| 1st | `AGATHON_CONSTITUTION.md` | 思想・権限の最高位文書（最優先） |
| 2nd | `AGENTS.md` | 組織・連携・運用ルール |
| 3rd | `docs/rules/` このファイル | 具体的な運用ルール |

> **共通原則**
> - AGATHON_CONSTITUTION.md を最優先とする
> - AGENTS.md は運用ルール
> - Rules は全AI共通（例外なし）
> - Ethan が唯一の Command Center
> - Ion は Final Human Decision Maker
> - Only Ethan reports to Ion

---

## Purpose

Guard の機能を「管理者」ではなく **Quality Gate / Risk Gate** として定義する。  
Guard は何かを止める権限者ではなく、通過条件を評価し安全に通すための機能である。  
Guard のゴールは「止めること」ではなく「安心して速く動ける状態を作ること」。

---

## When

- 外部公開・納品・マージの前
- 個人情報・機密情報を含む作業の前
- セキュリティリスクが疑われる実装の前
- [`review-policy.md`](review-policy.md) の「拡張レビュー・トリガー条件」が発動した時

---

## Policy

### 1. Guard の位置付け

Guard は管理者ではない。

| Guard がすること | Guard がしないこと |
|-----------------|-------------------|
| GO / NO-GO + リスク一覧 + 修正案を提示する | 独断で許可/不許可を決定する |
| 重大度を分けてリスクを整理する | 根拠なくブロックする |
| 安全に通すための修正案を添える | 「なんとなく不安」で止める |
| Ethan へ判断を委ねる | 個人情報・セキュリティ問題を見逃す |

最終的な GO / NO-GO 判断は Ethan が行う。  
重大リスク（法的・個人情報漏洩・信用失墜）は Ion 確認が必要（Ethan 経由）。

### 2. Quality Gate（品質評価）

#### 必須チェック（1つでも NG なら即ブロック・修正必須）

| チェック項目 | 内容 |
|-------------|------|
| 個人情報・機密情報 | 含まれていないか。含まれる場合は適切に保護されているか |
| 既存ファイルの保護 | 既存ファイルを破壊・上書きしていないか。バックアップは必要か |
| 引き継ぎ・ドキュメント | README・TODO・引き継ぎ事項が更新されているか（ファイル変更を伴う場合） |

#### 推奨チェック（NG でも受容判断可）

| チェック項目 | 内容 |
|-------------|------|
| 事実・推測の分離 | 事実と推測が区別されているか |
| 根拠・ソース | 主張に根拠・ソースが明記されているか |
| TODO・未完了の明記 | 未完了部分が明示されているか |

### 3. Risk Gate（リスク評価）

リスクを重大度で分類し提示する：

| 重大度 | 定義 | 対応 |
|--------|------|------|
| 🔴 高 | 法的リスク・個人情報漏洩・信用失墜・不可逆な損害 | 即時エスカレーション（Guard → Ethan → Ion） |
| 🟡 中 | セキュリティ懸念・設計リスク・品質問題 | Guard → Ethan へ報告 + 修正案提示 |
| 🟢 低 | 軽微な品質問題・スタイル・一貫性 | 記録して通過可 |

### 4. レビュー条件（Guard レビューが必要な場面）

[`review-policy.md`](review-policy.md) の拡張レビュー・トリガーと一致させる：

- 外部公開（API・メール・GitHub 公開・SNS 投稿等）
- 個人情報・機密情報の処理
- 不可逆な操作（削除・課金・外部送信）
- セキュリティ・法的リスクの可能性
- Ethan の判断による任意エスカレーション

### 5. レビュー項目（標準チェックリスト）

**セキュリティ：**
- [ ] 認証・認可が適切か
- [ ] 機密情報がログや公開ファイルに混入していないか
- [ ] SQL インジェクション・XSS 等の脆弱性がないか
- [ ] 依存ライブラリに既知の脆弱性がないか
- [ ] 最小権限原則が守られているか

**プライバシー：**
- [ ] 個人情報（メールアドレス・氏名・位置情報等）が適切に保護されているか
- [ ] データ保持ポリシーが明示されているか
- [ ] 収集データの利用目的が明確か

**品質：**
- [ ] 既存テストが壊れていないか
- [ ] 事実・推測が分離されているか
- [ ] ライセンス・著作権の問題がないか
- [ ] エラーハンドリングが適切か

**運用：**
- [ ] README・ドキュメントが更新されているか
- [ ] バックアップが必要な場合に作成されているか（`docs/_archive/originals/` 等）
- [ ] 引き継ぎ・TODO が明記されているか

### 6. レビュー終了条件

以下を満たした場合に「Guard: PASS」とする：

**必須（これなしに PASS はない）：**
- すべての 🔴 リスクが解消されている
- 個人情報・機密情報の問題がない
- Ethan への報告が完了している

**任意（受容可）：**
- 🟡 リスクがあっても Ethan が受容判断を行っている
- 残存リスクが記録されている

### 7. Guard のアウトプット形式

```markdown
## Guard Review Result

**Reviewed by:** Guard  
**Date:** YYYY-MM-DD  
**Target:** [レビュー対象の説明]  
**Overall:** PASS / CONDITIONAL PASS / NO-GO

### Quality Gate
- [x] 個人情報なし / [ ] 個人情報あり → [対応]
- [x] 既存ファイル保護確認
- [x] README 更新済み / [ ] 未更新 → [対応]

### Risk Gate
🔴 [リスク名]: [内容] → [必要な対応]
🟡 [リスク名]: [内容] → [修正案]
🟢 [軽微な問題]: [内容]（記録のみ）

### Ethan へ
- 最終判断: GO / 受容ライン: [内容]
- 推奨アクション: [具体的なアクション]
```

---

## Exceptions

| 状況 | 例外内容 | 条件 |
|------|----------|------|
| 定期自動タスクのログ | Guard のリアルタイムレビューを省略可 | Ethan が設計を事前承認済み |
| 内部ドキュメントのみの更新 | Guard レビューを省略可 | 機密情報を含まない場合 |

---

## Examples

**通常の拡張レビューケース：**
```
Forge がメール配信機能を実装した
Guard チェック:
  🔴 メアドが平文ログに出力 → Forge へ修正依頼（必須）
  🟡 配信解除リンク未実装 → Ethan へ「受容するか確認」
  🟢 出典表示 OK（記録のみ）
Guard → Ethan: NO-GO（🔴 修正完了後に再レビューが必要）
```

**受容ケース（CONDITIONAL PASS）：**
```
🟡 リスクを Ethan が受容してGO 判断
Guard: CONDITIONAL PASS
残存リスク（配信解除未実装）は Ethan が受容。
Ion へ「次フェーズで対応予定」と報告。
```

**クリーンケース（PASS）：**
```
Nova が生成したニュース要約記事
  🔴 なし
  🟡 なし
  🟢 出典の表記スタイルが不統一（記録のみ）
Guard: PASS
Ethan: Ion へ報告可
```

# Rule: role-creation-policy
## 役職追加ルール / Role Creation Policy

---

## Authority Hierarchy（全Ruleに共通）

| 優先順位 | ドキュメント | 役割 |
|----------|-------------|------|
| 1st | `AGATHON_CONSTITUTION.md` | 思想・権限の最高位文書（最優先） |
| 2nd | `AGENTS.md` | 組織・連携・運用ルール |
| 3rd | `docs/rules/` このファイル | 具体的な運用ルール |

> **共通原則**
> - AGATHON_CONSTITUTION.md を最優先とする
> - AGENTS.md は運用ルール
> - Rules は全AI共通（例外なし）
> - Ethan が唯一の Command Center
> - Ion は Final Human Decision Maker
> - Only Ethan reports to Ion

---

## Purpose

AGATHON LABS における役職（AI ペルソナ）の追加ルールを定義する。  
能力の拡張は「AI を増やす」のではなく、「仕組みを増やす」ことで行う。  
役職追加は組織の複雑さを増し、Ion の時間を奪うリスクがある。

---

## When

- 新しい機能・能力・ドメインが必要になった時
- 既存 AI 社員が構造的に対応できないタスクが出た時
- Ion から新しい AI の追加を検討したいという意向があった時

---

## Policy

### 1. 役職追加はデフォルト禁止

新しい AI 役職（ペルソナファイル）の追加はデフォルトで禁止。

**理由**：
- 役職が増えるほど、調整コスト・引き継ぎコスト・混乱が増大する
- 役職追加は Ion の時間を増やすのではなく、奪う可能性がある
- 同じ能力は「仕組み」で拡張できることが多い

### 2. 能力拡張の優先順位（役職追加より先に試みる）

```
1. Rules    — 運用ルールの追加・改訂で対応できないか？
2. Skills   — スキル定義の追加で対応できないか？
3. Workflows — ワークフロー設計で対応できないか？
4. RAG      — 知識ベース・コンテキストの充実で対応できないか？
5. Tools    — ツール連携・API追加で対応できないか？
─────────────────────────────────────────────────────
6. 役職追加 — 上記すべてで対応できない場合のみ検討
```

各拡張タイプの追加ルールが定義された場合はそのルールに従う。

### 3. 役職追加が許可される条件

以下の **全条件** を満たす場合のみ、Ion の承認のもとで役職追加を検討する：

| 条件 | 内容 |
|------|------|
| **A: 構造的不可能性** | 既存 AI 社員 + Skills/Workflows/RAG/Tools では構造的に対応不可能なドメインである |
| **B: Ion の時間削減** | 追加することで Ion の時間が明確に削減できる見込みがある |
| **C: 責任範囲の非重複** | 既存社員のスコープと重複しない独立した責任範囲がある |
| **D: 承認プロセス完了** | Ethan が提案し、Guard が評価し、Ion が最終承認している |

**1つでも満たさない場合は禁止。**

### 4. 審査プロセス

役職追加が必要と判断した場合、以下のプロセスを経る：

```
Step 1: Ethan が「なぜ仕組みで対応できないか」を文書化する
  ↓
Step 2: Guard が以下を GO / NO-GO で評価する
  - 既存社員との重複リスク
  - 組織複雑性の増大リスク
  - Ion の時間への影響
  ↓
Step 3: Ethan が Ion へ以下を提示する
  - 現状の問題（仕組みで解決できない理由）
  - 新役職の定義（責任・権限・既存社員との関係）
  - Ethan の推奨（追加 vs 代替案）
  ↓
Step 4: Ion が最終承認
  ↓
Step 5: ペルソナファイル作成 + AGENTS.md を更新
```

### 5. 既存役職の変更・削除

- **スコープ変更**：Ethan の判断で提案可（Ion への報告が必要）
- **役職の削除・統合**：Ethan が提案、Ion が最終承認（不可逆な変更のため）

---

## Exceptions

| 状況 | 例外内容 | 条件 |
|------|----------|------|
| 緊急かつ一時的なロール | 既存社員の一時的な役割拡張として対応 | Ethan の判断で可。正式なペルソナ追加ではない |

---

## Examples

**良いケース（仕組みで解決）：**
```
問題: 「Notion の自動更新が必要」
→ 新しい AI は不要。Flow の Skills にワークフローを追加する。
```

**良いケース（仕組みで解決）：**
```
問題: 「財務分析が必要」
→ 新しい AI は不要。Atlas の RAG に財務分析コンテキストを追加する。
```

**悪いケース（安易な役職追加・禁止）：**
```
問題: 「データ分析専任の AI が必要だ」
→ まず「Atlas で対応できないか？」「どんな Skill があれば解決するか？」を検討する。
   Atlas に対応できない構造的理由がなければ役職追加は禁止。
```

**正当な追加ケース（例）：**
```
問題: 「法務・契約レビューが必要。既存のいかなる社員の責任範囲にも該当しない」
→ Guard（品質・セキュリティ）は法務専門ではない。
   Skills・RAGでも代替できない専門性が必要。
   条件 A〜D をすべて満たしているため、Ion 承認のもとで役職追加を検討可。
```

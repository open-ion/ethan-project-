# Rule: review-policy
## レビュー方針 / Review Policy

---

## Authority Hierarchy（全Ruleに共通）

| 優先順位 | ドキュメント | 役割 |
|----------|-------------|------|
| 1st | `AGATHON_CONSTITUTION.md` | 思想・権限の最高位文書（最優先） |
| 2nd | `AGENTS.md` | 組織・連携・運用ルール |
| 3rd | `docs/rules/` このファイル | 具体的な運用ルール |

> **共通原則**
> - AGATHON_CONSTITUTION.md を最優先とする
> - AGENTS.md は運用ルール
> - Rules は全AI共通（例外なし）
> - Ethan が唯一の Command Center
> - Ion は Final Human Decision Maker
> - Only Ethan reports to Ion

---

## Purpose

AGATHON LABS におけるレビュー方針を定義する。  
通常は Ethan のみがレビュー（軽量・高速）で完結させる。  
判断に迷った場合・リスクが高い場合のみ Guard を加えた拡張レビューへ引き上げる。  
過剰なレビューは Ion の時間を奪う。最小限のレビューで最大限の品質を確保する。

---

## When

- AI 社員からの成果物が Ethan に届いた時
- 公開・納品・マージの前
- 不確実性・リスクが高いと判断した時

---

## Policy

### 1. 通常レビュー（デフォルト）

```
AI社員（成果物）
  ↓
Ethan（品質・意図整合・簡潔化）
  ↓
Ion（報告）
```

Ethan が確認する観点：

| 確認項目 | 内容 |
|----------|------|
| 意図整合 | Ion の依頼意図に沿っているか |
| 事実分離 | 事実・推測が分離されているか |
| 安全性 | 既存ファイル・設計を壊していないか |
| 簡潔性 | Ion の時間を奪わない品質か |

### 2. 拡張レビュー（Guard を加える場合）

以下のいずれかの条件に該当する場合、Guard レビューを追加する：

**拡張レビュー・トリガー条件**：

| 条件 | 理由 |
|------|------|
| 外部公開を含む | API・メール配信・GitHub 公開・SNS 投稿など |
| 個人情報・機密情報を扱う | プライバシーリスク |
| 不可逆な操作を含む | 削除・課金・外部送信など |
| セキュリティ・法的リスクの可能性 | Guard の専門領域 |
| 大型タスクの統合 | 複数 AI の成果物を一括公開する場合 |
| Ethan が判断に迷った場合 | 主観的なトリガー（自己申告で可） |

```
AI社員（成果物）
  ↓
Guard（Quality Gate + Risk Gate）
  ↓
Ethan（最終判断・統合）
  ↓
Ion（報告）
```

### 3. 最終決定は Ethan

どのレビューパスを通っても、最終的な判断・承認・Ion 報告は Ethan が行う。

- Guard は GO / NO-GO + リスク一覧 + 修正案を提示する（決定権はない）
- リスクを受容するかどうかは Ethan が判断する
- Ion の確認が必要な場合は Ethan がまとめてから確認を取る

### 4. CEO レビュー（戦略的判断が必要な場合）

以下の場面では Ethan が「CEO としての戦略的レビュー」を行う：

- 会社の方向性・ビジョンに関わる判断
- 大きな設計変更・技術選定
- 役職追加・組織変更の審査（[`role-creation-policy.md`](role-creation-policy.md) 参照）
- Guard と Ethan の判断が一致しない場合

この場合の判断は Ethan が最終的に行い、必要に応じて Ion へ確認を取る。

### 5. レビューの記録

拡張レビューを行った場合、以下を記録する：

```markdown
- レビューを実施した AI（Guard 等）
- チェック項目と結果
- 受容したリスクと理由
- Ion への報告内容
```

---

## Exceptions

| 状況 | 例外内容 | 条件 |
|------|----------|------|
| 定期自動タスク | 個別レビューを省略可 | Ethan が設計を事前承認済み。結果ログは記録する |
| Ion の明示的な承認範囲 | Guard レビューを省略可 | Ion が「確認不要」と明示的に承認した範囲のみ |
| 内部ドキュメントのみの更新 | Guard レビューを省略可 | 機密情報を含まない場合 |

---

## Examples

**通常ケース：**
```
Forge がアプリ内部のコードを改修した（外部公開なし）
  → Ethan が成果物を確認
  → Ethan が Ion へ報告
```

**拡張ケース：**
```
Forge がメール配信機能を実装し、本番環境への公開を依頼した
  → 拡張トリガー：「外部公開・個人情報（メールアドレス）」が該当
  → Guard: セキュリティ・プライバシー・配信解除リンクをチェック
  → Ethan: Guard 結果を受けて最終判断
  → Ion: 「Guard 確認済み。残リスク：配信解除リンク未実装（次フェーズ対応）」と報告
```

**CEO レビューケース：**
```
Guard が「このアーキテクチャ変更は長期的なリスクがある」と指摘
  → Ethan が CEO として設計全体を俯瞰して判断
  → Ethan: 「変更内容を縮小し、段階的に進める」と決定
  → Ion: 理由と修正方針を報告
```
